plr-prompt-lab — 실습 가이드 (인수인계)

PLR(객체 속성 추출) 프롬프트를 측정하며 개선하는 독립 실험 도구. 아래를 순서대로 그대로 복붙하면 모든 기능을 GPU 없이 체험합니다 — §7(실측)만 GPU가 필요합니다. 구조 시각화는 STRUCTURE.html. (2026-07-03 · plr_v1.5_cot 강제커밋 · 모든 명령 실행 검증됨)

∞ 전체 개선 루프 — 이 도구의 존재 이유

┌────────────────────────────────────────────────────────────────────────┐
│  ① 데이터셋 준비          ② 기준선 측정            ③ 오답 분석          │
│  crops + labels + manifest │ run (모델 1회, GPU)     │ gallery 오답 필터   │
│  validate-dataset          │ eval -A all             │ raw_responses.jsonl │
│         │                  │ └→ gallery/report 자동  │ /improve-prompt 스킬│
│         ▼                  ▼                         ▼                   │
│  ⑥ 승격 ◄─── ⑤ A/B 판정 ◄─── ④ 프롬프트 새 버전 작성                    │
│  PROMPT_VERSION bump       │ experiment (구 vs 신)   │ prompts/<신버전>/  │
│  lab port → 배포(재시작)   │ report 비교표           │ (author-prompt 계약)│
│                            └── 지면 ④로 다시 ────────┘                   │
└────────────────────────────────────────────────────────────────────────┘
실행 순서 요약 (섹션 번호 = 아래 상세):
  1. 준비 — cp -r examples/dataset_template datasets/내이름 → 크롭·라벨 채우기 → lab validate-dataset (§5)
  2. 기준선 — lab run -X plr_v1.5_cot --dataset datasets/내이름 (모델 크롭당 1회) → lab eval -A all --dataset datasets/내이름 — gallery/report가 자동 생성됨 (§5, §5⁺)
  3. 분석 — gallery.html의 오답 필터(AND/OR)로 실패 패턴 찾기, 모델 원문은 raw_responses.jsonl, 체계적 개선안은 skills/improve-prompt 6역할 루프 (§2, §8)
  4. 새 버전 — cp -r prompts/plr_v1.5_cot prompts/plr_v1.6_시도 → 문구 수정 (계약: unknown 금지·enum 주입 유지) → mock으로 배관 확인 (§6)
  5. A/B — experiment yaml에 prompts: [구버전, 신버전] → lab experiment run → report 비교표에서 Δ 확인. 지면 3~4 반복 (§4, §7)
  6. 승격 — 이겼을 때만 1회: skills/co-change 승격 체크리스트 (byte-sync → 양쪽 테스트 → PROMPT_VERSION bump) → 운영 반영은 컨테이너 재시작 (§9)
②까지는 mock으로 GPU 없이 연습 가능 — 실측(§7)만 GPU가 필요합니다.

1 demo — 전체 루프를 3초에 체험 (mock 모델)

python3 lab.py demo --keep    # --keep: 생성물(datasets/demo/)을 지우지 않고 남김
기대 결과: 합성 데이터셋 생성 → mock 재채점 2회 → 평가 → 두 버전의 Δ 출력. datasets/demo/에 crops/·labels.jsonl·predictions.jsonl·raw_responses.jsonl·ledger.jsonl·manifest.yaml이 남습니다.
MockModel이란: GPU 없이 미리 정해진 PLR YAML을 돌려주는 결정적 가짜 모델. 파이프라인 배관(재채점→파싱→평가→리포트)을 모델 없이 검증하는 용도이며, 프롬프트 품질과는 무관합니다 — 품질 측정은 §7의 실측으로.
자동 렌더링: §2·§3의 두 HTML은 lab eval·lab experiment가 끝날 때 자동 생성됩니다(데이터셋마다 gallery.html, ledger 옆 report.html — --no-render로 끔). 아래 수동 명령은 옵션을 바꿔 재생성할 때 씁니다.

2 gallery — 크롭을 눈으로 검수

python3 lab.py gallery --dataset datasets/demo
[gallery] written: datasets/demo/gallery.html
브라우저로 열면: 오답 우선 정렬, CORRECT/WRONG 배지, margin·quality 표시, 클래스 필터. 다속성 데이터셋이면 카드마다 속성별 태그가 붙고 속성 체크박스 + AND/OR 오답 필터가 생깁니다 (§5⁺ 참고).

3 report — ledger를 비교표·히트맵으로

python3 lab.py report --ledger datasets/demo/ledger.jsonl --out report.html
[report] wrote report.html (2 record(s), …)
전 실험 비교표·추이·confusion. 실험군 간 비교는 --compare 다른ledger.jsonl.

4 experiment — 조합 매트릭스 한 방에

python3 lab.py experiment run examples/experiment.example.yaml
[experiment] total=2  ok=2  failed=0
datasets × models × prompts × attributes 교차곱을 셀 단위로 실행하고 ledger에 적재. yaml 안의 상대경로는 yaml 파일 기준으로 해석됩니다. 종료코드 0=전부 성공 / 2=전부 실패.
예제 yaml을 열어 축을 바꿔보세요: models: [mock]→[gemma]가 실측 전환, prompts:에 버전을 추가하면 A/B가 됩니다. 스펙: docs/EXPERIMENT_SPEC.md

5 내 데이터셋 만들기 — 다속성 라벨

cp -r examples/dataset_template datasets/my_test            # ① 템플릿 복사 (다속성 예제)
# ② crops/<obj_id>.jpg 채우고 manifest.yaml의 attributes: 맵 선언
# ③ labels.jsonl — 크롭당 한 줄, 속성별 정답을 dict로:
#    {"obj_id":"M3","labels":{"gender":"male","helmet":"helmet"}}
python3 lab.py validate-dataset --dataset datasets/my_test  # ④ 형식 검증 (fail-loud)
python3 lab.py run -X plr_v1.5_cot --dataset datasets/my_test              # ⑤ 모델 1회 — -A 불필요 (mock 가능)
python3 lab.py eval -A all --dataset datasets/my_test       # ⑥ 라벨 있는 전 속성 한 번에 채점 ★
모델 호출은 크롭당 1회 — run이 attributes.jsonl에 전체 plr_json을 저장하고, eval -A all이 속성별 예측을 pred_path로 재추출해 채점합니다 (GPU 재실행 없음). gender / vehicle_type / military는 내장 프리셋 (빈 dict로 선언), 커스텀 속성은 labels: + pred_path: 선언. 라벨 정책: 판별 불가 = unknown(채점 제외·별도 집계), 그 속성 키 자체를 빼면 미라벨(조인에서 제외). 단일 label 형식(구 데이터셋)도 계속 동작. gallery는 --attribute로 속성을 골라 배지를 그립니다. 사람·차량이 섞인 데이터셋은 행에 "object_type": "vehicle"을 적으면 그 크롭만 차량 프롬프트로 나갑니다. 상세: docs/DATASET_SPEC.md, 스킬: skills/prepare-dataset/SKILL.md

5⁺ 속성(-A) 레시피 — 경우의 수 전부

run — 모델은 속성 무관, -A는 predictions.jsonl 추출 뷰만 
python3 lab.py run -X plr_v1.5_cot --dataset datasets/my_test               # 표준: -A 생략 → manifest 첫 속성
[run] --attribute 생략 → manifest 첫 속성 'gender' 사용 (추출 뷰만 결정; 전체 속성은 attributes.jsonl에 저장됨)
python3 lab.py run -X plr_v1.5_cot --dataset datasets/my_test -A helmet     # 추출 뷰를 명시하고 싶을 때
python3 lab.py run -X plr_v1.5_cot -A gender                                # --dataset 생략 시엔 -A 필수 (eval/golden/<A> 규약 경로)
eval — 단일 / 쉼표 나열 / all 
python3 lab.py eval -A gender --dataset datasets/my_test          # 하나만 (라벨 고친 속성만 재채점할 때)
python3 lab.py eval -A gender,helmet --dataset datasets/my_test   # 골라서 — 나열 순서대로, 각각 ledger 1줄
python3 lab.py eval -A all --dataset datasets/my_test             # 라벨이 실제로 있는 속성 전부
all의 동작: 선언됐는데 라벨 없는 속성은 실패가 아니라 skip+안내(exit 0) — [eval] 라벨 없음 → skip: military (…). 라벨에는 있는데 manifest 미선언이면 WARNING(오타 방지). manifest 선언이 아예 없으면 라벨 키에서 직접 도출. 나열 중 하나가 실패해도 나머지는 계속 (마지막 요약 + exit 1).
gallery — 기본은 전부, 속성별 태그 + AND/OR 오답 필터 
python3 lab.py gallery --dataset datasets/my_test                 # 라벨된 속성 전부 — 카드마다 속성별 태그
python3 lab.py gallery --dataset datasets/my_test -A gender,helmet # 부분 집합만
python3 lab.py gallery --dataset datasets/my_test -A helmet       # 하나만 → 단일 배지 화면
다속성 화면의 필터바: [전체|오답만|정답만] × 속성 체크박스 × [OR(하나라도)|AND(모두)]. 예) "오답만 + gender·helmet 체크 + AND" = 두 속성 다 틀린 크롭만. 정렬은 오답 많은 카드 먼저, 저-margin 순. 실물 예시: gallery_multi_example.html
사람·차량 혼합 데이터셋 — 행 단위 object_type이 프롬프트를 라우팅 
# labels.jsonl — 크롭마다 종을 적으면 run 한 번에 각자 맞는 프롬프트가 나간다:
{"obj_id": "p1", "object_type": "person",  "labels": {"gender": "female"}}
{"obj_id": "v1", "object_type": "vehicle", "labels": {"vehicle_type": "sedan"}}

python3 lab.py run -X plr_v1.5_cot --dataset datasets/mixed   # p1→person 프롬프트, v1→vehicle 프롬프트
python3 lab.py eval -A all --dataset datasets/mixed           # gender는 사람 크롭에서, vehicle_type은 차량 크롭에서
부분 라벨 — 세 상태의 구분 
{"obj_id": "a", "labels": {"gender": "male", "helmet": "helmet"}}    # 둘 다 채점 참여
{"obj_id": "b", "labels": {"gender": "unknown", "helmet": "helmet"}} # gender: 사람도 판별불가 → 채점 제외+별도 집계
{"obj_id": "c", "labels": {"helmet": "helmet"}}                      # gender: 미라벨 → 조인에서 자연 제외 (아직 안 단 것)

prompts/plr_v1.5_cot/          # 운영 기본 버전 (기능별 파일 5개가 한 세트)
    person.yaml                # system + user_cot(운영) + user_plain + commit_enums
    vehicle.yaml  query_parser.yaml  vqa.yaml  retry.yaml

cp -r prompts/plr_v1.5_cot prompts/plr_v1.6_test   # ① 복사해서 새 버전 (이름 ≤16자!)
# ② person.yaml의 문구를 수정 — {colors} 같은 주입 자리는 절대 지우지 말 것
python3 lab.py run --version plr_v1.6_test --dataset datasets/my_test --model mock  # ③ 배관 확인
지켜야 할 계약(위반 시 테스트가 빨간불): unknown을 선택지로 제시 금지(저신뢰는 margins로) · enum은 {colors} 자리에 자동 주입(손글씨 금지) · 출력 형식은 파서와 한 몸(슬롯 추가 시 skills/co-change의 매트릭스 따라 동행 수정). 전체 계약: skills/author-prompt/SKILL.md. 샘플링 등 knob 조합 실험은 configs/example.yaml 복사 → lab run --version <config명>.

7 실측 (여기부터 GPU)

# 운영 ir 컨테이너가 GPU 점유 중 — 관리자와 협의 후 중지하고:
python3 lab.py run  --version plr_v1.5_cot --dataset eval/golden/gender  # --model gemma가 기본
python3 lab.py eval --attribute gender --dataset eval/golden/gender
# 준비된 A/B (v1.4 vs v1.5 강제커밋 — labels.jsonl 라벨 완료가 전제):
python3 lab.py experiment run examples/gender_ab_v14_vs_v15.yaml
run은 크롭당 모델 1회 — predictions.jsonl(예측·margin·quality)과 raw_responses.jsonl(모델 원문·입출력 토큰 수)을 남깁니다. eval은 acc/recall/precision/F1/confusion/bias + margin·quality 캘리브레이션을 ledger에 적재.
주의 ① 실 CCTV 크롭은 사적 데이터 — git에 커밋 금지 (eval/golden/*/crops는 .gitignore 확인). ② 운영 ziosummary-ir 컨테이너 재시작은 배포 행위(프롬프트 버전 교체+재인덱싱 유발) — 임의로 하지 말 것. ③ GPU 사용 후 ir 컨테이너 재기동 여부는 관리자 결정.

8 개선 루프 — 스킬과 함께

오답이 보이면 skills/improve-prompt/SKILL.md: 분석자 → 제안자 → 비판자 ⇄ 수정자 → 판단자(최대 3라운드). 분석자는 오답 크롭을 이미지로 직접 열람, 모든 제안은 본 obj_id + 수치 근거 필수. 산출물 = prompts/<신버전>/ 초안 + 검증용 experiment yaml → §4로 측정 → §3으로 비교.

개발 중에는 core/ir 없이 lab만으로 완결됩니다. 실험 승자가 확정되면 core/ir 보유자가 skills/co-change/SKILL.md의 승격 단계 체크리스트(바이트 동기화 → 양쪽 테스트 → PROMPT_VERSION bump)를 1회 수행 — 그 전까지 동기화하지 마세요.
