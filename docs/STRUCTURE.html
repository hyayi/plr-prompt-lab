4 프롬프트 2-소스 구조 — 이 프로젝트의 핵심

"모델에게 실제로 나가는 프롬프트는 무엇인가?"
두 갈래 — 여기가 여러 번 헷갈렸던 지점 (yaml만 고쳐서는 기본 프롬프트가 안 바뀜)
▼
① 기본 경로 — plr_prompts.py 하드코딩 상수
--version 없음 / yaml 없는 버전(mock 등) → 상수 사용
= 현재 v1.4 · core/ir과 바이트 동일 (이식 대상)
IR_PLR_FORMAT = yaml|json IR_PLR_REASON = on|off
env가 어느 상수를 쓸지 선택 → experiment의 formats:/reasons: 축이 이걸 스윕
② 버전 경로 — prompts/<V>.yaml
lab run --version V → FilePromptProvider가 로드
plr 블록 — v1.5는 commit_enums: true로 unknown 없는 enum 주입 (구버전 yaml은 역사 보존)
yaml의 format: 키가 wire format 고정 IR_PLR_REASON은 양쪽 공통
▼
주입구 (build_messages=None — 하위호환 seam)
plr_core.run_plr(build_messages=…) — PLR 프롬프트
None이면 상수 경로와 바이트 동일 (운영 동작 보존)
▼
model.generate(messages, image)
gemma (GPU, lazy load) / mock (GPU-free) — registry.py에서 선택
▼drift 방어 — 두 소스가 어긋나는 순간을 잡는 3중 게이트
TESTtests/test_prompt_surface_parity.py — 상수 ↔ yaml 자동 parity 게이트 (+ query_parser 블록) DIFFlab port — lab ↔ core/ir 표면 diff (읽기전용, --apply는 사람) HASHprovenance.prompt_hash — ledger에 프롬프트 상태 각인
plr-prompt-lab · STRUCTURE.html (자체완결 · 외부 의존성 없음) — GUIDE.html과 함께 배포 가능
