PLR Prompt Lab · 구조 & 사용법

plr-prompt-lab

PLR(객체 속성 추출)과 텍스트 검색의 프롬프트·파이프라인을 측정하며 개선하는 독립 실험 도구. 데이터·모델·프롬프트·파이프라인을 파라미터로 조합해 실험하고, 자동 평가 결과를 추이로 추적하며, 한 장의 HTML 리포트로 시각화한다.

🎯 핵심 원칙 — “측정 없이 프롬프트를 바꾸지 않는다.” 개선안은 골든셋으로 검증(eval)한 뒤에만 운영 코드(core/ir)로 의도적으로 이식(port)한다.
무엇인가 전체 흐름 구조 파일 지도 4개 파라미터 데이터 형식 사용법 ledger & 리포트 전달·이식

01무엇인가

운영 서비스와 완전히 분리된, 프롬프트 개선 전용 샌드박스.

🔬 측정 우선

고정된 골든셋(정답 라벨)에 프롬프트를 돌려 정확도·편향·recall을 숫자로 잰다. 성별 프롬프트가 v0.5↔v0.7 사이에서 편향이 진동한 이력 — 재기 전엔 방향을 모른다.

🧩 조합 실험

데이터 × 모델 × 프롬프트 × 파이프라인의 교차곱을 한 번에 실행. 어떤 조합이 나은지 매트릭스로 비교.

📦 전달 가능

운영 코드에서 개발 표면만 lean 추출한 별도 저장소. DB·Redis·서비스 없이, 받는 사람이 GPU만 있으면 독립 실행.

두 개의 개발 대상(파이프라인)
파이프라인하는 일측정 지표
plr크롭 이미지 → 프롬프트 → 모델 → 파싱 → 속성 판별(성별/차종/군용)accuracy · 혼동행렬 · bias
search텍스트 쿼리 → 파싱 → 하드필터/스코어 → 랭킹recall@k · precision@k

02전체 흐름 (한 사이클)

파라미터를 고르고 → 실행하면 → 자동 평가되어 → ledger에 쌓이고 → 리포트로 시각화된다.

plr 셀은 모델을 호출해 예측을 만들고, search 셀은 그 속성으로 랭킹을 낸다. 두 경로 모두 운영 코드와 같은 공유 코어(run_plr / run_search)를 쓰므로, 여기서 이긴 프롬프트는 lab port diff로 그대로 운영에 옮길 수 있다.

03구조 (레이어)

모델은 얇은 인터페이스로 분리 — 프롬프트·파이프라인이 특정 모델에 얽히지 않는다.

모델 Model 인터페이스

Model.generate(messages, image) 하나로 추상화. 구현: gemma(실제 GGUF·GPU), mock(결정론·GPU-free, 데모/테스트). 갈아끼우기 쉽게.

파이프라인 공유 코어

run_plr()(크롭→속성) · run_search()(쿼리→랭킹). core/ir에서 lean 추출한 바로 그 함수 — parity 보장.

데이터 Dataset

디렉터리 하나 = 한 데이터셋(crops/+labels.jsonl+queries.jsonl+manifest.yaml). --dataset로 선택.

평가 eval 루프

run_eval(accuracy/bias) · run_search_eval(recall@k). 직전 버전 대비 Δ를 계산해 ledger.jsonl에 append.

조합 registry + experiment

registry.py가 모델/파이프라인을 이름으로 관리. experiment.py가 experiment.yaml의 교차곱을 실행.

시각화 report

report.py가 ledger를 읽어 자체완결 HTML(inline SVG) 생성 — 서버·CDN 불필요.

운영(core/ir)과의 관계 — 이 랩은 운영 코드의 복제가 아니라 개발 표면만 lean 추출한 것이다. DB·Redis·인덱싱 서비스는 가져오지 않는다(랩은 DB에 절대 접근하지 않음). 개선은 lab port가 보여주는 diff를 사람이 확인하고 eval 통과 후 운영에 적용한다.

054개 파라미터

이 네 축을 조합해 실험한다.

축플래그값 예시
데이터셋--dataset <경로>받는 사람이 만든 골든셋 디렉터리
모델--modelgemma (실측·GPU) · mock (연습·GPU-free)
프롬프트--versionplr_v1.3_cot · plr_v1.4_cot …
파이프라인--pipelineplr (속성) · search (검색)

06데이터 형식 (받는 사람이 만드는 것)

데이터셋 = 디렉터리 하나. 자세한 스키마는 DATASET_SPEC.md.

# 데이터셋 디렉터리 구조
my_dataset/
├── crops/
│   ├── 1001.jpg      # 파일명 = obj_id
│   ├── 1002.jpg
│   └── ...
├── labels.jsonl      # 사람 정답
├── queries.jsonl     # (검색용, 선택)
└── manifest.yaml     # attribute·n·생성정보
# labels.jsonl (한 줄 = 한 객체)
{"obj_id":"1001","label":"female"}
{"obj_id":"1002","label":"male"}

# queries.jsonl (검색 정답)
{"query":"검은색 차","relevant":["1002"]}

# manifest.yaml
attribute: gender
n: 63
검증은 필수 — python3 lab.py validate-dataset --dataset my_dataset 가 PASS 할 때까지 형식을 맞춘다(크롭↔라벨 정합, 필수 필드 등을 fail-loud로 알려줌). 데이터셋 구성이 처음이면 prepare-dataset 스킬을 Claude Code에 올려 대화형으로 진행.

07사용법 (단계별)

실제 명령 그대로. 실측(모델 호출)은 GPU가 필요하고, 연습은 mock으로 GPU 없이.

  1. 설치

    파이썬 가상환경 + 의존성. 실측하려면 Gemma GGUF 모델 다운로드가 필요(자세히는 INSTALL.md).

    python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt      # llama-cpp-python은 GPU 빌드

  2. 루프 체득 — 데모 (GPU 불필요)

    합성 데이터 + mock 모델로 build→run→eval 한 사이클을 즉시 재현. 도구가 어떻게 도는지 먼저 눈으로 확인.

    python3 lab.py demo

  3. 데이터셋 준비 & 검증

    영상 인덱스에서 크롭을 뽑고(build-golden) 오분류만 지정해 라벨링(label), 혹은 직접 구성 후 검증.

    python3 lab.py build-golden --video <video_id> --attribute gender --dataset my_dataset
python3 lab.py label --dataset my_dataset --female-in-male M3,M7 --male-in-female F2
python3 lab.py validate-dataset --dataset my_dataset   # → PASS 까지

  4. 단일 실험 — 실행 & 평가

    바꾼 프롬프트를 골든 크롭에만 재실행(재색인 없음)하고 직전 버전 대비 Δ를 본다.

    python3 lab.py run  --model gemma --pipeline plr --version plr_v1.4_cot --attribute gender --dataset my_dataset
python3 lab.py eval --attribute gender --dataset my_dataset
# → accuracy 0.90  Δ vs plr_v1.3_cot: +0.11   bias female→male: 0.12

  5. 조합 실험 — 매트릭스

    experiment.yaml에 축을 나열하면 교차곱 전 셀을 run→자동 eval→ledger로 실행. 한 셀이 실패해도 나머지는 계속(fail-loud-but-continue). 스키마는 EXPERIMENT_SPEC.md.

    # experiment.yaml
datasets:  [./my_dataset]
models:    [gemma]
prompts:   [plr_v1.3_cot, plr_v1.4_cot]   # 2개 → 2셀 비교
pipelines: [plr]
attributes:[gender]

python3 lab.py experiment run experiment.yaml

  6. 시각화 — 리포트

    ledger를 한 장의 자체완결 HTML로. 브라우저로 더블클릭하면 오프라인에서 열린다.

    python3 lab.py report --out report.html
# 추이 차트 · 모델×프롬프트 히트맵 · "어떤 프롬프트 변경 → 어떤 지표 Δ"

  7. 이식 — 개선을 운영으로

    이긴 프롬프트를 운영 코드와 diff. 기본은 읽기 전용(운영 미변경). 실제 적용은 사람이 --apply로 확정하고 parity 테스트를 돌린다.

    python3 lab.py port                 # lab ↔ core/ir 프롬프트 표면 diff (읽기전용)
python3 lab.py port --apply         # 확정 적용 (검증 후)

08ledger & 리포트

모든 실험은 append-only 기록으로 남아 추이를 만든다.

ledger 레코드 한 줄

{
  "attribute":"gender", "version":"plr_v1.4_cot",
  "accuracy":0.90, "bias":{"pair":"female->male","rate":0.12},
  // 조합 키 (무엇이 이 결과를 냈나)
  "dataset":"my_dataset", "model":"gemma",
  "pipeline":"plr", "prompt_hash":"744125fa"
}

리포트 3개 섹션

📈 추이

버전 진행에 따른 accuracy·bias·recall 라인.

🔥 히트맵

모델 × 프롬프트 지표 그리드 — 조합 비교.

🔀 프롬프트 변화 → Δ

연속 버전의 prompt_hash 전이와 그 지표 변화를 연결.

prompt_hash는 실제 프롬프트 표면(prompts/*.yaml + plr_prompts.py + plr_core.py)의 내용 해시 — lab port가 이식하는 바로 그 파일들과 동일한 단일 출처.

09전달 · 이식 (외부 엔지니어용)

프롬프트를 개선해 되돌리는 사람을 위한 요약. 자세히는 HANDOFF.md.

✏️ 편집

prompts/*.yaml만 고친다. 운영 추론·저장 코드는 건드리지 않는다.

📏 측정

고치면 반드시 run→eval로 Δ를 확인. 악화면 롤백.

↩️ 되돌리기

이긴 프롬프트는 lab port로 diff를 떠서 ZioVision 팀에 전달. 팀이 eval 게이트 후 운영 적용.

실측 전제 — run이 Gemma를 로드하므로 GPU + 모델이 필요하다(운영 ir과 GPU 경합을 피해 전용/오프피크 사용). 라벨링은 사람 정답이 필요하다. GPU가 없으면 --model mock·lab demo로 워크플로만 익힌다.
